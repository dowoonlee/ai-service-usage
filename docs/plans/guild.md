# 길드 (트레이너 그룹) — 기획

> 랭킹 사용자(트레이너)의 상위 개념 '길드' 도입. **기획 문서 — 실제 구현 전.**
> 작성 시점: v0.13.15 기준.

## 0. 결론 요약

랭킹 옵트인 사용자들이 **코인으로 길드 생성권을 구매**해 길드를 만들고, **초대 코드로 결성**하고, **월간 길드 랭킹**(멤버 상위 5명 VP 합산)으로 경쟁한다. 월간 Top3 길드는 기여 멤버 전원이 RP를 받는다. 기존 인프라(HMAC 인증, `rp_rewards` 트랙, lazy finalize RPC, `claim-reward` 라우팅, 소모품 티켓 패턴)를 최대한 재사용해 서버 신규 개념을 최소화한다.

## 1. 확정 결정

| 항목 | 결정 | 근거 |
|---|---|---|
| 이름 | **길드** (코드/DB: `guild`) | 트레이너 세계관과 일관. SwiftUI `Group`, SQL `GROUPS`, Claude `organization`, 펫 `party` 충돌 회피 |
| 랭킹 산식 | **월간 VP 상위 5명 합산** | 소규모 길드도 경쟁 가능, 유령 멤버·부계정 부스팅 이득 제한 |
| 최대 인원 | **제한 없음** | 운영 단순화. 점수는 상위 5명만 반영되므로 대형 길드의 점수 이점은 구조적으로 제한됨 |
| 가입 방식 | **초대 코드만** (MVP) | 지인/팀 단위 사용에 부합, 서버 복잡도 최소. 공개 목록·검색은 Phase 3 |
| 생성 비용 | **길드 생성권 (3,000 코인, 소모품)** | 코인 sink 추가 + 잡길드 난립 억제. 프리미엄 가챠권(`premiumTickets`)과 동일한 소모품 패턴 |
| 보상 | **월간 Top3 길드 → 기여 멤버 전원 RP** — 1위 500 / 2위 300 / 3위 200 | 개인 월간 정산(1000/600/400)의 절반 스케일. 기존 `rp_rewards` 트랙 재사용. §4 악용 방지의 자격 조건 필수 |
| 이적 쿨다운 | **7일** | 월말 용병 영입 완화. 서버 정책 상수로 조정 가능 |
| 길드장 액션 API | **`guild-manage` 하나로 통합** (kick/코드 재발급/해체) | 함수 수 억제. 1함수 1액션 관례의 예외로 명시 |
| 1인 길드 | **리더보드 노출** | 점수는 1인분 그대로 — 숨길 이유 없음. 혼자서도 시작할 수 있어 진입 장벽 낮춤 |
| UI 위치 | **가챠 창 전용 "길드" 탭** (6번째 탭, 랭킹 오른쪽) | 상점/파티/도장/레포트/랭킹과 동급 표면. 랭킹 탭 내 세그먼트 안(초안)보다 발견성이 좋고 화면이 넉넉함 |
| UI 중심 | **길드 사무실** — 멤버들이 자기 대표 펫을 사무실 스팟(자리)에 배치, 펫들이 돌아다니는 공유 공간 | 단순 리스트보다 "같이 있는" 체감. 픽셀 펫 자산·애니메이션 인프라 재사용 |

## 2. 규칙 상세

### 길드 생성권 (코인 소모품)
- 길드 생성에는 **생성권 1장**이 필요하다. 생성권은 **3,000 코인**으로 구매 (가격은 릴리스 전 조정 가능 — 10연차 2,700, 파티 슬롯 첫 구매 2,000보다 약간 높은 "의미 있는 결단" 가격).
- 모델은 프리미엄 가챠권과 동일한 소모품: `Settings.guildPermits` 보유 수량, `CoinLedger` 경유 구매, `IntegrityGuard` 체크섬 대상에 포함.
- **소모 시점 = 서버 `guild-create` 성공 응답 후.** 서버 실패 시 생성권은 차감되지 않는다 (가챠 roll/commit 분리와 같은 원칙 — 사용자에게 유리한 쪽으로).
- 해체 후 재생성에도 새 생성권이 필요하다 (영구 권리 아님).
- 코인은 로컬 경제이므로 서버는 생성권을 검증하지 않는다 — 생성권은 클라이언트 게이트이고, 서버 측 실질 가드는 생성 rate-limit(§4). 로컬 상태 조작은 기존과 동일하게 `IntegrityGuard` 탐지 → `abuse_flags` 보고.
- 구매 UI는 길드 탭의 창설 카드에 통합 ("생성권 구매 → 길드 만들기" 원스톱). 상점 탭 노출은 P3에서 검토.

### 생성/가입/탈퇴
- 랭킹 등록(옵트인) 사용자만 생성/가입 가능 — 게시판과 동일 전제.
- **1인 1길드.** 생성자는 길드장이 된다.
- 길드명: 유니크(대소문자 무시), 길이·금칙어는 닉네임 validation(`_shared/validation.ts`) 재사용.
- 초대 코드: 서버 발급 영숫자 8자리. 길드장이 **재발급(rotate)** 가능 — 코드 유출 대응.
- 탈퇴/추방 후 **7일 가입 쿨다운** — 월말 고득점 유저 이적으로 점수 뻥튀기(용병 시장) 완화. 값은 서버 정책 상수로 두고 조정 가능하게.

### 길드장
- 권한: 멤버 추방(kick), 초대 코드 재발급, 길드 해체.
- 길드장 탈퇴/계정 삭제 시: 가입일 순 최고참에게 **자동 승계**. 마지막 1인 탈퇴 시 자동 해체.

### 사무실 (공유 공간)
- 길드마다 픽셀 아트 **사무실** 씬이 있고, 멤버는 자기 **대표 펫**(랭킹 프로필의 대표 펫 — 서버가 이미 보유)을 사무실의 **스팟(자리)** 하나에 배치한다.
- 스팟은 dev 밈으로 명명된 고정 슬롯 **12개**: 창가 책상, 스탠딩 데스크, 서버랙 옆, 커피머신 앞, 소파, 화이트보드 앞, 옥상(?) 등. 이름은 구현 시 확정.
- **선착순 점유** — 이미 다른 멤버가 쓰는 스팟은 선택 불가. 언제든 빈 스팟으로 이동 가능(본인 자리만). 배치는 선택 사항 — 미배치 멤버는 멤버 리스트에만 표시.
- 인원 무제한 대비: 스팟 12개 초과 멤버는 사무실에 안 보이고 리스트에만 존재. 스팟 수 확장(증축)은 P3 후보.
- 펫은 자기 스팟 주변을 돌아다니고(walk/idle/sit), 랜덤 대사 말풍선(기존 `Quotes` 재사용). **이번 달 활동 반영**: 상위 5명(점수 기여자) 펫은 자리에서 "일하는" 모션 + ★, 그 달 VP 0인 멤버 펫은 잠자는 모션 — 사무실만 봐도 길드 상태가 읽히게.
- Mythic 오라·이로치 색조 등 기존 펫 연출 그대로 적용.

### 랭킹/정산
- 길드 월간 점수 = **그 달 멤버별 VP 중 상위 5명의 합** (멤버가 5명 미만이면 있는 만큼).
- 집계 대상 멤버 = 조회/정산 시점의 현재 멤버 (가입 시점 무관 — 단순화. 이적 악용은 쿨다운으로 완화).
- 월 마감 정산은 기존 패턴 그대로: `guild-leaderboard` 호출 시 `finalize_monthly_guild_rp_if_needed` RPC lazy 트리거 (별도 cron 없음).
- 개인 랭킹과 완전 독립 — 개인 VP/RP 정산에는 어떤 영향도 없음.

### 보상 (RP)
- 월간 Top3 길드의 **자격 멤버 전원**에게 RP 지급.
- **자격 조건 (필수)**: finalize 시점 멤버이면서 **해당 월 VP > 0**. 인원 무제한 + 전원 보상 조합에서 "초대 코드만 받아 무임승차" 악용을 막는 최소 가드.
- 지급액: 1위 500 / 2위 300 / 3위 200 RP (확정 — 개인 월간의 절반 스케일).
- 수령은 기존 `claim-reward` 확장: `rp_rewards`에 길드 트랙(period type `guild-monthly`) 추가, dedup 동일 패턴.

## 3. 서버 설계

### 스키마 (migrations)

```sql
guilds (
  id              uuid PRIMARY KEY,
  name            text NOT NULL,          -- unique lower(name)
  invite_code     text NOT NULL UNIQUE,
  leader_device_id text NOT NULL,         -- FK → 랭킹 계정
  created_at      timestamptz
)

guild_members (
  guild_id    uuid REFERENCES guilds ON DELETE CASCADE,
  device_id   text UNIQUE,                -- UNIQUE = 1인 1길드
  joined_at   timestamptz,
  office_slot smallint,                   -- 사무실 스팟 index (0..11), NULL = 미배치
  PRIMARY KEY (guild_id, device_id),
  UNIQUE (guild_id, office_slot)          -- 스팟 선착순 점유
)

guild_join_cooldowns (device_id, until)   -- 탈퇴/추방 시 7일 기록

-- 월간 점수는 뷰로: 멤버별 월간 VP에 guild별 row_number() ≤ 5 필터 후 SUM
-- 정산 결과는 rp_rewards 재사용 (period_type 'guild-monthly' 추가) 또는 guild_monthly_winners
```

- `device_id` 비교는 전부 lowercase 정규화 (기존 board #94 교훈).
- 계정 삭제(`delete` 함수): `guild_members` CASCADE + 길드장 승계 처리 추가.

### Edge Functions (기존 "함수당 액션" 스타일)

| 함수 | 역할 | 인증 |
|---|---|---|
| `guild-create` | 생성 (이름 검증, 코드 발급, 생성 rate-limit) | HMAC |
| `guild-join` | 초대 코드로 가입 (쿨다운·1인1길드 검사) | HMAC |
| `guild-leave` | 탈퇴 (승계/자동 해체 처리) | HMAC |
| `guild-manage` | 길드장 액션 묶음: kick / 코드 재발급 / 해체 | HMAC |
| `guild-info` | 내 길드 상세 (멤버 목록 + 이번 달 기여 VP + 각자 대표 펫·office_slot) | HMAC |
| `guild-office` | 내 스팟 선택/이동/비우기 (스팟 점유 충돌은 409) | HMAC |
| `guild-leaderboard` | 길드 월간 랭킹 + 직전 달 Top3 + pending 보상. finalize RPC lazy 트리거 | HMAC |

- 신규 7개 (총 27개). 길드장 액션 3종(kick/코드 재발급/해체)은 `guild-manage` 하나로 통합 (확정) — 기존 1함수 1액션 관례의 의도적 예외. `guild-office`는 본인 스팟 self-액션이라 leader 전용 `guild-manage`와 분리.
- 멤버 대표 펫은 랭킹 프로필(리더보드/시상대가 이미 쓰는 대표 펫)을 재사용 — 신규 저장 없음. `guild-info`가 join해서 내려준다.
- 정책 상수(쿨다운 일수, 코드 길이, 보상액)는 `_shared/guild_policy.ts`로 SSOT화 — `board_policy.ts` 패턴.

## 4. 악용 방지

| 위협 | 대응 |
|---|---|
| 무임승차 (코드만 받고 Top3 보상 수령) | 보상 자격 = 해당 월 VP > 0 (§2 보상) |
| 월말 용병 영입 | 탈퇴/추방 후 7일 가입 쿨다운 |
| 부계정 부스팅 | 상위 5명 산식 자체가 상한. 개인 VP 조작은 기존 submit abuse cap + `IntegrityGuard` abuse_flags 그대로 |
| 길드 생성 스팸 | 생성권 3,000 코인(경제적 억제) + `register_attempts` 패턴 재사용 (IP/device 단위 rate-limit, 서버 측 실질 가드) |
| 생성권 로컬 조작 | 코인/티켓과 동일 — `IntegrityGuard` 체크섬 대상 포함, 위반 시 abuse_flags 보고 (탐지 전용) |
| 길드명 어뷰징 | 닉네임 validation 재사용 + 유니크 제약 |

## 5. 클라이언트 설계

- **`RankingAPI` 확장**: guild 6개 endpoint + 모델(`GuildInfo`, `GuildMember`, `GuildLeaderboardEntry`). HMAC 서명은 기존 헬퍼 그대로.
- **`Settings`**: `guildId` / `guildName` / `isGuildLeader` 캐시 (표시용 — 서버가 SSOT) + **`guildPermits`** (생성권 보유 수, `IntegrityGuard` 대상).
- **`CoinLedger`**: `purchaseGuildPermit()` — 파티 슬롯 구매(`purchasePartyPresetSlot`)와 동일 패턴. 정책 상수 `guildPermitCost = 3_000`.
- **UI**: 가챠 창 6번째 탭 `GuildView.swift` 신규 — 상세는 §5-1.
- **보상 수령**: `ViewModel.checkPodiumRewardIfNeeded()`에 길드 pending 보상 트랙 추가 (기존 이중 트랙 → 삼중 트랙).
- **알림**: `NotificationManager`에 길드 시상대 보상 케이스 추가.
- **트레이너 카드**: 길드명 태그 표시 (랭킹 행 호버·기여자 카드에도 자동 반영).

### 5-1. UI 설계 — 길드 탭 (`GuildView.swift`)

**탭 추가**: `GachaView.Tab`에 `case guild` 추가, 순서는 `상점 · 파티 · 도장 · 레포트 · 랭킹 · 길드` (랭킹의 사회적 확장이므로 랭킹 오른쪽 끝). 6분할 segmented picker는 520pt 폭에서 칸당 ~85pt — 2글자 라벨이라 여유 있음. `.gachaSwitchTab` NotificationCenter 라우팅도 그대로 동작.

**뷰 골격**: `RankingView`와 동일한 패턴 — 헤더 + 게이트 + 콘텐츠, `GachaView`가 frame 제어(560×640), 탭 열려 있는 동안 5분 자동 새로고침.

**상태 머신** (4-state, 위에서부터 순서대로 게이트):

| 상태 | 조건 | 화면 |
|---|---|---|
| 빌드 미구성 | `!RankingAPI.isConfigured` | placeholder (랭킹 탭과 동일 문구) |
| 랭킹 미등록 | `!settings.rankingRegistered` | "길드는 랭킹 참여자 전용입니다. 설정 → 랭킹에서 참여를 시작하세요." + 설정 열기 버튼 |
| 미가입 | `guildId == nil` | **온보딩 화면** (아래) |

- 구매·창설·가입 모두 확인 alert (10연차 확인과 동일 스타일).
- 가입 쿨다운 중이면 가입 카드에 "재가입까지 N일" 안내 + 버튼 비활성.

**가입 (내 길드) 레이아웃** — 사무실 씬이 화면의 주인공, 나머지는 아래 스크롤:

### 5-2. 사무실 씬 기술 설계

**렌더링** — `GuildOfficeView` (SwiftUI ZStack):

- 배경: 픽셀 사무실 배경 1장 + 스팟별 가구 오버레이. 씬은 3개의 가로 레인(바닥선)으로 구성, 각 스팟은 (레인, x구간) 고정 좌표.
- 펫: `WalkingCat`을 그대로 쓰지 않는다 — 차트(plotFrame/points)에 강결합. 대신 **`OfficePetController` 신규** (경량, ~150줄 예상): 아래 공간·충돌·행동 모델을 따르는 상태 기계. 스프라이트 프레임 재생·이로치 색조·Mythic 오라는 `PetSprite`/`PetEffectOverlay`의 기존 경로 재사용.
- 최대 12마리 동시 애니메이션: 메뉴바(30Hz 단일 합성)와 달리 SwiftUI TimelineView 기반으로 충분 — 가챠 창은 열려 있을 때만 렌더.
- 대사: 기존 `Quotes` 풀 재사용 + 사무실 전용 대사 소량 추가 ("커피 리필하러 감", "빌드 기다리는 중" 등 — `Quotes.swift`에 office 카테고리).

**공간·좌표 모델** — `OfficeLayout` (코드 상수, 정적 정의):

- 논리 캔버스 **280×150** (@2x 렌더 = 560×300pt). 위 0~60은 벽 밴드(장식 전용), 60~150이 바닥(활동 영역).
- **레인 3개** (baseline y = 84 / 112 / 140). 각 레인에 스팟 4개 = 12자리. 뒷레인은 벽걸이·창가류(화이트보드 앞, 창가 책상), 중간 레인은 데스크 클러스터·커피머신, 앞레인은 소파·라운지.
- 스팟 정의 = `(id, 이름, lane, anchorX, 가구 구성)`. 가구는 스팟에 종속된 정적 오브젝트 — 레이아웃은 전 길드 공통 1종(고정 배치)으로 시작, 꾸미기는 P3.
- **z-order = baseline y 오름차순** (뒷레인부터 그림). 같은 레인에서는 가구 먼저, 펫 나중 — 펫이 가구 "앞"을 지나가는 구도. 레인이 다르면 앞레인 펫이 뒷레인 가구를 자연스럽게 가린다.

**충돌 모델**:

| 대상 | 충돌 | 규칙 |
|---|---|---|
| 바닥/벽 배경 | 없음 | 바닥은 자유 통행. 씬 좌우 경계(x 0~280)와 벽 밴드 하단(y=60)만 이동 클램프 |
| **가구** | **있음 (AABB footprint)** | 가구마다 baseline 기준 blocking 폭을 정의 (스프라이트 폭보다 약간 좁게 — 픽셀아트 관례). **같은 레인에서만 blocking** — 다른 레인의 펫은 원근상 앞/뒤로 지나가는 것이므로 통과. footprint는 `OfficeLayout`에 하드코딩 (2dPig manifest.json의 footprint 값을 초기값으로 활용) |
| **펫 ↔ 펫** | **없음 (통과)** | 하드 충돌은 12마리가 좁은 레인에서 서로 막혀 교착·떨림을 만들고 회피 로직 비용만 커짐 (코지 심 장르 관례도 비충돌). 대신 **스침 상호작용**: 같은 레인에서 두 펫이 교차하는 순간 낮은 확률(~10%)로 둘 다 1.5초 멈춤 + 인사 말풍선("👋", "커피?", "머지 좀…") — 펫별 30초 쿨다운으로 도배 방지 |

- 충돌 판정 구현: 레인별로 가구 footprint를 **blocked x-interval 목록으로 사전 계산** — 펫 이동은 1D(x축)라 판정이 구간 비교 하나로 끝난다. 경로탐색(A*) 불필요.
- 이동 중 막히면: 목표를 포기하고 **방향 반전** (roomba 방식). 레인 내 walkable 구간이 가구로 분할되므로, 목표 x는 항상 자기 스팟이 속한 구간 안에서만 샘플링 → 애초에 막힐 일이 드물다.

**행동 상태 기계** (`OfficePetController`):

```text
idleAtSpot ──(3~10s)──▶ wanderNearSpot ──(도착/막힘)──▶ idleAtSpot
    │                        │
    │ (상위 5명)             │ (교차 시 ~10%)
    ▼                        ▼
 working (자리 고정,      greeting (1.5s 멈춤 + 말풍선)
  PC ON 프레임 연동)          │
    │                        ▼
    │ (VP 0 멤버)         wanderNearSpot 복귀
    ▼
 sleeping (소파/자리 고정, 💤 — wander 안 함)
```

- `wanderNearSpot`: 스팟 anchorX ± 40px 범위 내 랜덤 목표, 자기 레인의 walkable 구간으로 클램프. 이동 속도는 WalkingCat 산책 속도와 동일 감각.
- `working`(상위 5명): 자리 고정 + 낮은 빈도 wander. 스팟 가구가 PC면 PC ON 애니 3프레임 연동.
- `sleeping`(월 VP 0): 완전 고정 + 💤. 스침 상호작용에도 반응하지 않음.
- 대사 타이머는 상태와 독립 (기존 WalkingCat quote 확률 모델 재사용, 말풍선은 씬 경계 클램프).
- 공용 지점 방문(커피머신으로 걸어가 커피 마시고 돌아오기 등 레인 간 이동)은 **P2** — MVP는 자기 레인·자기 스팟 주변으로 한정해 충돌 케이스를 최소화.

**에셋** — 신규 필요 항목이자 이 기능의 주요 리스크:

- 필요: 사무실 배경(벽/바닥) + 가구 12종 내외 (책상·모니터·커피머신·소파·서버랙·화이트보드·책장·창문 등). 픽셀 톤은 기존 팩(16~32px)과 맞출 것.
- 조달: **P0 리서치 + 검증 완료 → `docs/research/office-assets.md`**. 1안 확정 = `2dPig Pixel Office Asset Pack`(CC0 원문 확인, 16px, GitHub 미러 검증 완료) — 요청 12종 중 9종 커버, 실물 목업으로 기존 펫 팩과의 화풍 조화 검증 통과. 공백 3종(서버랙·창문·스탠딩데스크)만 코드 드로잉. PC는 ON 애니 3프레임이 있어 "작업 중" 모션에 바로 사용.
- SwiftPM 리소스 flatten 제약(전 리소스 basename 유니크) 그대로 적용.

**동기화**:

- 배치 상태는 `guild-info` 응답에 포함 (member별 office_slot + 대표 펫 kind/variant). 별도 실시간 채널 없음 — 5분 자동 새로고침 + 수동 ↻ 로 충분 (동시 편집 충돌은 스팟 UNIQUE 제약 + 409로 해소).
- 내 스팟 변경은 낙관적 업데이트 (즉시 이동 → 실패 시 롤백 + 토스트).
